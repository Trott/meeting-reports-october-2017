Hi, everybody! This year, I got to attend Node.js Interactive. Or, actually, it's Node interactive, because, like it says there, the js is silent.
Let's start with what Node is. Node is a JavaScript runtime that uses an event-driven, non-blocking I/O model that... No, just kidding. That means nothing.

Let's try this. Let's talk about how we use Node at the UCSF Library. None of our big projects are implemented primarily in Node. However, almost all of them have some significant Node component. Node is very versatile.

IDL Solr Proxy

IDL uses a Node.js proxy to protect it's search index.

Ilios LTI calendar

Ilios implemented the new LTI calendar feature using Node.js.

Library hours

The Library website uses a Node.js API middleware program so that we have more flexibility in displaying the hours on the website than Springshare gives us by default.

EAD -> Calisphere

And, a new one: we use Node.js to pre-process XML finding aids for import into Nuxeo.

Atwood's Law

There are other examples, but the thing is this: Even if you don't think you're using Node.js, you're probably using Node.js. If you use the Slack desktop app, that's a Node.js app. Jeff Atwood, a software engineer and writer, proposed Atwood's Law: Any application that can be written in JavaScript will eventually be written in JavaScript. And he was basically talking about Node.

JIFASNIF

Because Node.js is a way to run JavaScript. JavaScript used to be a browser-only language. Node.js is a way to bring it pretty much anywhere else. The acronym up there means "JavaScript is fun and so Node is fun."

So back to the event: There were multiple keynotes that addressed diversity and inclusion directly or indirectly.

Kim Crayton's entire keynote focused on D+I illusions, truths we don't want to face, and major barriers. Look it up. It's totally worth watching.

Open Source Sustainability

Panel including Maggie Pint + Erin McKean

There was a panel on open source sustainability. Maggie Pint commented that people often focus on how to direct money to open source developers and small open source projects, but that is not what these people and projects need. "We need accommodation from our employers to fold our open source projects into our day-to-day lives." I'm grateful to have that opportunity here.

And Erin McKean noted that "Having work accommodation [to work on open source] improves diversity [in open source] because it's not just privileged people who have the spare time to contribute." And in listing off the many benefits of having someone employed who works in open source there was this gem: "As we've seen with Equifax, just because you're using open source doesn't mean you are using open source well. Having employees who are involved in open source might help in those situations."

Data Sciences Workshop

Much to my surprise, the Data Sciences workshop was standing room only. Node.js has many strengths, but data science is not generally considered one of them. People tend to gravitate to Python or R instead. But there are people working to change that. And they're doing a great job.

DatProject.org

Related, by the way, is the Dat Project, which is a data sharing project written in Node.js. They're coordinating with UC, and I posted something to the UCSF Library Slack about it some time ago. They've visited Davis and Merced. Not sure what kind of traction they're getting but I'm definitely keeping an eye on it.

hospitalrun.io

Another Node.js project of note that was the subject of a talk: HospitalRun is free, easy-to-us software for developing world healthcare. It does patient records, business records, etc. I took a *lot* of notes at this talk. I hope we can do a hack day in the CKM with it some time soon. It's aligned with the university's mission and the underlying technology looks a lot like us: Node.js of course, but also Ember.js (used by Ilios) and ElasticSearch (used by IDL). We have the domain expertise and we have the technological expertise. We should be able to contribute.
